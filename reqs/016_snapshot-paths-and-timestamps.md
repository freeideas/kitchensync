# 016_snapshot-paths-and-timestamps: Snapshot path hashing and timestamp conventions

## Behavior
This concern derives from `specs/database.md` sections "Path Hashing" and "Timestamps", plus `specs/sync.md` sections "TMP Staging" and "BAK Directory" where the same timestamp format is used for peer-side staging and archive directories. It covers relative path hash input normalization, xxHash64/base62 encoded identifiers, root and parent hash conventions, timestamp string format, UTC and microsecond precision, lexicographic ordering, per-process monotonic timestamp generation, and the cases where copied deletion estimates are intentionally not fresh generated timestamps.

## $REQ_IDs
- `016.1` -- Snapshot path identifiers are the xxHash64 seed-0 digest of the entry's full relative path encoded as an 11-character zero-padded base62 string using digits `0-9`, uppercase `A-Z`, and lowercase `a-z`.
- `016.2` -- Snapshot path identifier input uses forward slashes and has no leading slash.
- `016.3` -- Snapshot path identifier input has no trailing slash.
- `016.4` -- Snapshot path identifiers for files and directories at the same relative path are computed from the same path input.
- `016.5` -- A snapshot row's parent identifier is the xxHash64 seed-0 digest of its parent directory's relative path encoded as an 11-character zero-padded base62 string using digits `0-9`, uppercase `A-Z`, and lowercase `a-z`.
- `016.6` -- Snapshot rows for entries directly under the sync root use the xxHash64 seed-0 digest of `/` encoded as an 11-character zero-padded base62 string as their parent identifier.
- `016.7` -- The sync root directory itself has no snapshot row.
- `016.8` -- Generated timestamps use the filesystem-safe format `YYYY-MM-DD_HH-mm-ss_ffffffZ`.
- `016.9` -- Generated timestamps represent UTC time.
- `016.10` -- Generated timestamps preserve microsecond precision.
- `016.11` -- Generated timestamps sort lexicographically in chronological order.
- `016.12` -- Snapshot database timestamp column values use the same `YYYY-MM-DD_HH-mm-ss_ffffffZ` timestamp format.
- `016.13` -- BAK directory timestamp path segments use the same `YYYY-MM-DD_HH-mm-ss_ffffffZ` timestamp format.
- `016.14` -- TMP directory timestamp path segments use the same `YYYY-MM-DD_HH-mm-ss_ffffffZ` timestamp format.
- `016.15` -- Log output timestamps use the same `YYYY-MM-DD_HH-mm-ss_ffffffZ` timestamp format.
- `016.16` -- Each generated timestamp value within one KitchenSync process is strictly greater than every timestamp value previously generated by that process.
- `016.17` -- Each `last_seen` write that needs a new current time stores a fresh generated timestamp instead of reusing a run-start timestamp.
- `016.18` -- Each BAK directory name that needs a timestamp uses a fresh generated timestamp instead of reusing a run-start timestamp.
- `016.19` -- Each TMP directory name that needs a timestamp uses a fresh generated timestamp instead of reusing a run-start timestamp.
- `016.20` -- A copied `deleted_time` value is allowed to equal another timestamp value from the same run.

## Notes
This category owns reusable identifier and timestamp value conventions. The static `snapshot` table schema that stores those values belongs to `005_snapshot-storage`; row mutation timing and deletion-estimate copying belong to `009_snapshot-updates`; physical BAK/TMP directory cleanup belongs to `011_displacement-and-staging-cleanup`.
