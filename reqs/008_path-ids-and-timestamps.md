# 008_path-ids-and-timestamps: Stable path IDs and sync timestamps

## Behavior
This concern derives from `specs/database.md` sections "Path Hashing" and
"Timestamps", plus `plan/path-ids.md` and `plan/local-file-metadata.md`. It
covers relative path hash inputs, xxHash64 seed and base62 encoding, parent ID
rules, root sentinel hashing, timestamp string format, UTC microsecond
precision, lexicographic ordering, process-local monotonic generation, and the
distinction between generated timestamps and copied deletion estimates.

## $REQ_IDs
- `008.1` -- Each snapshot row `id` is the zero-padded 11-character base62 encoding of the xxHash64 seed-0 value for that entry's full path relative to the sync root.
- `008.2` -- Path ID base62 strings use the alphabet `0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz`.
- `008.3` -- Snapshot path IDs hash relative paths with forward slash separators, no leading slash, and no trailing slash.
- `008.4` -- Snapshot path IDs use the same relative path string rule for files and directories.
- `008.5` -- A snapshot row's `parent_id` is the zero-padded 11-character base62 encoding of the xxHash64 seed-0 value for the row's parent directory relative path.
- `008.6` -- Snapshot rows for entries directly under the sync root use `JyBskcNRrBK` as `parent_id`.
- `008.7` -- The sync root directory itself has no snapshot row.
- `008.8` -- Timestamp strings in database columns, BAK directory names, TMP directory names, and log output use the format `YYYY-MM-DD_HH-mm-ss_ffffffZ`.
- `008.9` -- Timestamp strings in database columns, BAK directory names, TMP directory names, and log output represent UTC times at microsecond precision.
- `008.10` -- Timestamp strings in database columns, BAK directory names, TMP directory names, and log output sort lexicographically in the same order as their represented UTC times.
- `008.11` -- Within one process, each generated timestamp value is strictly greater than every timestamp value previously generated by that process.
- `008.12` -- Within one sync run, generated `last_seen` timestamp values are not reused across snapshot rows.
- `008.13` -- When an entry is confirmed absent or displaced, its `deleted_time` is copied from that row's existing `last_seen` value.
- `008.14` -- Descendant rows affected by a displacement cascade use the displaced entry's deletion estimate as their `deleted_time`.

## Notes
This file covers representation and generation. Where timestamps are stored or
used for cleanup belongs to the snapshot, staging, and logging categories that
own those behaviors.
