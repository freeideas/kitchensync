# 02_snapshot-db: Per-peer snapshot database

## Behavior

Each peer keeps its own snapshot of what it has had at `{peer-root}/.kitchensync/snapshot.db`. At the start of a run, each peer's snapshot is downloaded to a local temp directory; all reads and writes happen against that local copy. After sync completes, each peer's snapshot is uploaded back via TMP staging and atomic rename. Derived from `database.md` and `sync.md` §Run.

## $REQ_IDs

- `02.18` — Each reachable peer ends a successful sync with a `.kitchensync/snapshot.db` SQLite file at its root.
- `02.20` — The snapshot file has one row per tracked entry, with columns recording at minimum `basename`, `mod_time`, `byte_size`, `last_seen`, and `deleted_time`.
- `02.21` — Directories are tracked in the snapshot with `byte_size = -1`; files have a non-negative byte count.
- `02.22` — The sync root directory itself has no row in the snapshot — only its descendants do.
- `02.23` — Timestamps stored in the snapshot use the format `YYYY-MM-DD_HH-mm-ss_ffffffZ` (UTC, microsecond precision).
- `02.24` — The snapshot upload is performed atomically: the new database lands at `.kitchensync/snapshot.db` only via a same-filesystem rename from a staging path under `.kitchensync/TMP/`.
- `02.25` — When a directory entry is decided to be deleted on a peer, every existing descendant snapshot row for that peer (rows where `deleted_time IS NULL` and which lie inside the displaced subtree by `parent_id` chain) receives a `deleted_time` value as part of the same snapshot update.
- `02.38` — A snapshot row's `id` column is the 11-character zero-padded base62 encoding of the xxHash64 (seed 0) of the entry's relative path; the `parent_id` column is the same encoding applied to the parent directory's relative path, with root-level entries using the hash of `/` as the sentinel `parent_id`.
- `02.40` — Timestamps generated by the program within a single process run are strictly monotonic: when a new timestamp value would collide with a previously generated one, 1μs is added so the new value is strictly greater.
- `02.41` — The snapshot schema defines exactly one table, named `snapshot` (singular, lowercase).
- `02.42` — The snapshot SQLite database is opened in rollback-journal mode with foreign-key enforcement enabled.
- `02.43` — Only `snapshot.db` is treated as sync state under `.kitchensync/`; sidecar SQLite files (e.g., `-shm`, `-wal`) are not synced.
- `02.45` — Every `last_seen` write, BAK timestamp directory, and TMP timestamp directory created within one process receives a freshly generated timestamp value that is strictly greater than all previously generated timestamp values in that process; copied `deleted_time` values are not timestamp-generator call sites.
- `02.49` — At startup, each reachable peer's `.kitchensync/snapshot.db` is copied to a local temporary path (`{tmp}/{uuid}/snapshot.db`), and snapshot reads and writes during the run operate on that local copy.
- `02.50` — If a reachable peer has no existing `.kitchensync/snapshot.db`, the run creates a new empty snapshot database locally for that peer instead of treating the missing file as an error.
- `02.51` — The `snapshot` table has indexes on `parent_id`, `last_seen`, and `deleted_time`.
- `02.52` — When an entry is confirmed present on a peer during traversal, that peer's snapshot row is upserted with the entry's current `mod_time` and `byte_size` and `deleted_time = NULL`.
- `02.53` — Snapshot path hashes are computed from relative paths written with forward slashes and no leading or trailing slash, regardless of the host OS path separator.
- `02.54` — The `snapshot.id` column is the primary key for the snapshot table.
- `02.55` — Snapshot path hashes use the base62 alphabet `0-9`, `A-Z`, then `a-z`.
- `02.56` — A snapshot row's `basename` stores only the entry's final path component, and `basename`, `mod_time`, and `byte_size` are non-null in every row.
- `02.57` — When a directory displacement cascades deletion to descendant snapshot rows, each descendant receives the displaced entry's deletion estimate rather than a newly generated timestamp.

## Notes

Tombstones (`deleted_time IS NOT NULL`) are written when a row's entry is confirmed absent — see `03_decision-rules.md`. Retention windows for tombstones and stale rows live in `04_retention.md`.
