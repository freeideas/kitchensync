# snapshot:

## Purpose

Own the per-peer SQLite snapshot database and the local working-copy lifecycle for that database. The module records what each peer has last confirmed for each relative path, exposes row mutation APIs used by traversal and copy completion, generates snapshot path identifiers and fresh sync timestamps, and moves snapshot databases between peer storage and local temporary files.

The snapshot module stores and mutates peer history. It does not choose sync winners, connect peer URLs, perform user-file replacement, schedule copy retries, or render progress.

## Responsibilities

- Define and create the peer snapshot database at `{peer-root}/.kitchensync/snapshot.db` as a SQLite rollback-journal database with exactly one non-internal table named `snapshot`, no views, and indexes on `parent_id`, `last_seen`, and `deleted_time`.
- Keep SQLite sidecar files out of peer state. Only `.kitchensync/snapshot.db` is downloaded, uploaded, and considered the peer's snapshot database.
- Enforce the `snapshot` table shape: `id TEXT PRIMARY KEY`, `parent_id TEXT`, `basename TEXT NOT NULL`, `mod_time TEXT NOT NULL`, `byte_size INTEGER NOT NULL`, `last_seen TEXT NULL`, and `deleted_time TEXT NULL`.
- Represent one tracked relative path per row. The sync root itself has no row; direct children of the root use the hash of `/` as their `parent_id`.
- Compute `id` and `parent_id` from validated slash-separated relative paths using xxHash64 seed 0 encoded as an 11-character zero-padded base62 string with digits, uppercase letters, and lowercase letters. File and directory rows for the same relative path use the same path hash; `byte_size = -1` distinguishes directories.
- Store file byte sizes as the file size in bytes and directory byte sizes as `-1`. Store row `mod_time`, `last_seen`, and `deleted_time` values only in `YYYY-MM-DD_HH-mm-ss_ffffffZ` format when present.
- Provide the process-wide timestamp generator used for snapshot `last_seen` writes and for callers that need fresh BAK or TMP timestamp path segments. Every generated timestamp in a process must be strictly greater than every timestamp previously generated by that process.
- Treat copied deletion estimates as copied values, not fresh generated timestamps. `deleted_time` values copied from existing `last_seen` or tombstone values do not need to be unique within a run.
- In a normal run, recover incomplete peer-side `.kitchensync/SWAP/snapshot.db/` state before downloading the peer snapshot and before deciding whether that peer has snapshot history.
- In `--dry-run`, skip peer-side snapshot SWAP recovery and download the live `.kitchensync/snapshot.db` as it exists at startup.
- Download each reachable peer's live `.kitchensync/snapshot.db` to a distinct local temporary `{tmp}/{uuid}/snapshot.db`. If the live database is not found, create a new empty local snapshot database and report that the peer had no snapshot history at startup.
- Serve all snapshot reads and writes during sync from the local temporary database for that peer.
- Upload updated local snapshot databases after all queued file copies have finished in normal runs. In `--dry-run`, leave the updated local temporary databases local and do not upload them to peers.
- Upload snapshots through `.kitchensync/SWAP/snapshot.db/` without relying on rename-over-existing: write and close `new`, rename an existing live `snapshot.db` to `old`, rename `new` to live `snapshot.db`, then delete `old`.
- Expose lookup APIs by relative path for the sync module to classify peer state. Snapshot rows may inform decisions, but this module must not decide the group outcome for a path.
- Expose an upsert-confirmed-present operation that records the supplied entry `mod_time` and `byte_size`, sets `last_seen` to a fresh current sync timestamp, and clears `deleted_time`.
- Expose an upsert-intended-copy operation for queued file destinations that records the winning entry `mod_time` and `byte_size`, clears `deleted_time`, and leaves any existing `last_seen` unchanged. For a first-time destination row, `last_seen` remains NULL.
- Expose a mark-copy-complete operation that, after a file copy succeeds, sets the destination row's `last_seen` to a fresh current sync timestamp.
- Expose a mark-absent operation that is idempotent: if an existing row has `deleted_time = NULL`, set `deleted_time` to that row's current `last_seen` and do not change `last_seen`; if `deleted_time` is already non-NULL, leave the row unchanged.
- Expose a mark-displaced operation for successful BAK displacement. It sets the displaced row's `deleted_time` from its previous `last_seen` and, for displaced directories, cascades that same deletion estimate through non-tombstone descendants reachable by `parent_id` in the same peer's database.
- Keep directory cascade scope local to one peer database. A cascade for one peer must not modify any other peer's snapshot database, and it must not mark rows whose only path from the displaced directory passes through an already tombstoned or already purged intermediate row.
- Leave existing rows unchanged when the corresponding filesystem operation fails. Failed directory creation, failed displacement, listing failure subtree exclusion, unreachable peers, excluded paths, and failed user-entry SWAP recovery must not cause snapshot row mutation for the affected peer and subtree.
- Support opportunistic stale-row cleanup without making correctness depend on cleanup finishing in the current run. Cleanup may remove tombstones older than `--keep-del-days` and may remove obsolete non-tombstone rows only when they no longer appear in any peer listing and have `last_seen` older than `--keep-del-days` or `last_seen = NULL`.
- Ensure opportunistic snapshot cleanup does not delay the first directory scan or first eligible file copy.

## Boundaries

- The module consumes connected peer sessions, transport handles, normalized transport error categories, relative paths, entry metadata, and retention settings supplied by root-owned contracts.
- The module exposes `SnapshotStore`-style APIs for snapshot lookup, local database creation/download, row mutation, stale-row cleanup, snapshot history reporting, and SWAP-based upload/recovery.
- The module reports recoverable snapshot download, recovery, and upload failures to the caller with enough context for runtime diagnostics, including the peer and normalized transport error category when available. Runtime owns stdout rendering.
- The module may use transport operations to read, write, rename, and delete `.kitchensync/snapshot.db` and `.kitchensync/SWAP/snapshot.db/{new,old}`. It must not mutate user data paths, user-file SWAP directories, BAK directories for user entries, or TMP staging.
- The module does not parse command-line peers, normalize peer URLs, select fallback URLs, or decide whether a connected peer is canon, subordinate, or contributing. It only reports whether a peer's live snapshot database existed at startup.
- The module does not build traversal entry sets, apply excludes, classify winners, resolve conflicts, decide deletion versus existence, or enqueue copies. It provides stored row data and mutation primitives to the sync module.
- The module does not execute user-file copy replacement, directory creation, displacement to BAK, user-entry SWAP recovery, or BAK/TMP retention cleanup. Operations owns those peer-side effects and calls snapshot mutation APIs only after effects succeed.
- The module does not own copy concurrency, retry accounting, progress display, verbosity filtering, process exit status, or final completion output.
- Concurrent KitchenSync runs against the same peer are not coordinated by this module. If overlapping runs upload snapshots, the last upload wins; correctness relies on later runs rediscovering any missing decisions from peer contents and snapshot history.

## Error Obligations

- Treat a missing live `.kitchensync/snapshot.db` during startup download as a nonfatal new-peer condition and create an empty local database for that peer.
- Treat snapshot recovery or download failures other than `not_found` as peer-level startup failures for the caller to log and exclude from the reachable set.
- Preserve startup ordering: normal runs must complete snapshot SWAP recovery before reading the live snapshot and before reporting whether snapshot history existed.
- Implement snapshot SWAP recovery exactly for the observed `old`, `new`, and live `snapshot.db` combinations:
  - `old` and live `snapshot.db` exist: delete `new` if present, then delete `old`.
  - `old` and `new` exist while live `snapshot.db` is missing: rename `new` to live `snapshot.db`, then delete `old`.
  - `old` exists while `new` and live `snapshot.db` are missing: rename `old` to live `snapshot.db`.
  - `new` and live `snapshot.db` exist while `old` is missing: delete `new`.
  - `new` exists while `old` and live `snapshot.db` are missing: rename `new` to live `snapshot.db`.
- On snapshot upload failure before SWAP `old` exists, keep any pre-existing live snapshot in place and leave cleanup of any `new` state to later startup recovery.
- On snapshot upload failure after SWAP `old` exists, leave the snapshot SWAP state in place for a later normal startup recovery.
- Never update local snapshot rows for excluded paths, unreachable peers, failed listing subtrees, failed inline operations, or peer-side recovery failures whose caller will treat the subtree as untrusted.
