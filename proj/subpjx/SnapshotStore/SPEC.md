# SnapshotStore:

## Purpose

SnapshotStore owns KitchenSync snapshot state. It manages each peer's
`.kitchensync/snapshot.db` file, the local temporary SQLite copy used during a
run, the exact `snapshot` table shape, stable path IDs, timestamp strings,
snapshot row updates, opportunistic snapshot row cleanup, and peer-side
snapshot SWAP replacement.

SnapshotStore is not the sync decision maker. It stores and mutates the facts
that other children use to classify files, detect deletions, record intended
copies, and write the final peer snapshot after all copy work has finished.

## Responsibilities

SnapshotStore exposes startup operations for reachable peers. In a normal run,
it first recovers peer-side `.kitchensync/SWAP/snapshot.db/` state for each
peer before deciding whether that peer has snapshot history. In dry-run, the
caller must not ask it to recover peer-side snapshot SWAP state or upload
snapshots. Snapshot SWAP recovery uses these paths under the peer root:

- live snapshot: `.kitchensync/snapshot.db`
- SWAP new: `.kitchensync/SWAP/snapshot.db/new`
- SWAP old: `.kitchensync/SWAP/snapshot.db/old`

Snapshot SWAP recovery has these outcomes:

- If `old` and live `snapshot.db` exist, leave live `snapshot.db` in place and
  delete `old` and any `new`.
- If `old` and `new` exist and live `snapshot.db` is missing, rename `new` to
  live `snapshot.db` and delete `old`.
- If `old` exists while `new` and live `snapshot.db` are both missing, rename
  `old` to live `snapshot.db`.
- If `new` and live `snapshot.db` exist while `old` is missing, leave live
  `snapshot.db` in place and delete `new`.
- If `new` exists while `old` and live `snapshot.db` are both missing, rename
  `new` to live `snapshot.db`.

If snapshot SWAP recovery fails for a peer during normal startup,
SnapshotStore reports that peer as unavailable for snapshot startup. If the
download of live `.kitchensync/snapshot.db` fails with any error other than not
found, it reports that peer as unavailable. If the live snapshot is not found,
it creates a new empty local snapshot database for that peer and reports that
the peer had no snapshot history at startup.

For every peer that remains available after startup snapshot handling,
SnapshotStore downloads the peer snapshot to a local temporary
`{tmp}/{uuid}/snapshot.db` file or creates that file when the peer had no live
snapshot. During the run, all snapshot reads and writes go through this local
temporary file. Peer-side SQLite sidecar files are never treated as snapshot
state and are never downloaded or uploaded.

SnapshotStore creates and maintains SQLite databases in rollback-journal mode.
Each database contains exactly one table named `snapshot`, contains no views,
and has exactly these columns:

- `id TEXT PRIMARY KEY`
- `parent_id TEXT`
- `basename TEXT NOT NULL`
- `mod_time TEXT NOT NULL`
- `byte_size INTEGER NOT NULL`
- `last_seen TEXT NULL`
- `deleted_time TEXT NULL`

The table has indexes on `parent_id`, `last_seen`, and `deleted_time`.
SnapshotStore must not add other tables, views, columns, or alternate snapshot
table names.

SnapshotStore exposes lookup and row mutation operations by slash-separated
relative path below the sync root. It never stores a row for the sync root
directory itself. Every stored row represents a path below the sync root, and
the table stores one row for each tracked path the peer has or has had.

For each row, SnapshotStore computes `id` as the zero-padded 11-character
base62 encoding of the xxHash64 seed-0 value for the entry's full relative
path. It computes `parent_id` the same way for the parent directory relative
path. Hash input paths use forward slashes, no leading slash, and no trailing
slash for both files and directories. Entries directly under the sync root use
`JyBskcNRrBK`, the path ID for `/`, as `parent_id`. Base62 uses exactly this
alphabet:

```text
0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz
```

`basename` is the final path component. `mod_time` is the entry modification
time last observed on that peer. File rows store their size in bytes as
`byte_size`; directory rows store `-1`. `last_seen` is set only when an entry
is confirmed present by listing, completed file copy, or completed directory
creation. `last_seen = NULL` represents a copy destination that has been
decided but not completed. `deleted_time = NULL` represents an entry that
exists. A non-NULL `deleted_time` marks a tombstone, and tombstone rows remain
in the table until cleanup removes them.

SnapshotStore exposes one process-local timestamp generator for database
columns, BAK directory names, TMP directory names, and log output. It returns
UTC microsecond strings in `YYYY-MM-DD_HH-mm-ss_ffffffZ` format. Returned
values sort lexicographically in the same order as their UTC times and are
strictly greater than every timestamp value previously generated by this
process. A run must call the generator separately for each new `last_seen`
write; generated `last_seen` values are not reused across snapshot rows.
`deleted_time` values for confirmed absence and displacement are copied
deletion estimates, not new generated timestamps.

SnapshotStore exposes these row update operations:

- Confirm present: upsert the row with the observed `mod_time`, observed
  `byte_size`, a new `last_seen` timestamp, and `deleted_time = NULL`.
- Confirm absent: if the row exists and `deleted_time` is NULL, set
  `deleted_time` to that row's current `last_seen` and leave `last_seen`
  unchanged. If the row is already a tombstone, leave it unchanged.
- Record intended file copy: upsert the destination row with the winning
  `mod_time`, winning `byte_size`, and `deleted_time = NULL`, while preserving
  the existing `last_seen`; if no row exists, insert `last_seen = NULL`.
- Complete file copy: set the destination file row's `last_seen` to a new
  timestamp after the queued copy succeeds.
- Complete directory creation: upsert the destination directory row with
  the directory `mod_time`, `byte_size = -1`, `deleted_time = NULL`, and a
  new `last_seen` timestamp after the directory creation succeeds.
- Complete displacement: after the peer has successfully moved the entry to
  `BAK/`, set the displaced row's `deleted_time` to its previous `last_seen`.

Failed directory creation and failed displacement do not change that peer's
existing snapshot row. If the app exits before a queued copy finishes, the
destination row remains with `deleted_time = NULL` and its old `last_seen`, or
`last_seen = NULL` for a first-time destination.

For completed directory displacement, SnapshotStore cascades the displaced
directory row's deletion estimate through that same peer's local snapshot
database. The cascade updates only non-tombstone rows reachable from the
displaced directory by following `parent_id` links, does not change already
tombstoned rows, and does not change rows outside the displaced subtree. When
the same directory is displaced on multiple peers, the caller invokes the
cascade once per affected peer, and each invocation updates only that peer's
snapshot database.

SnapshotStore exposes opportunistic cleanup for old snapshot rows. Cleanup
removes tombstone rows whose `deleted_time` is older than `--keep-del-days`
days. It also removes obsolete non-tombstone orphan rows that cannot be reached
by a directory displacement cascade after their `last_seen` is older than
`--keep-del-days` days. Cleanup is maintenance work only: sync decisions must
be able to proceed without waiting for cleanup to finish in the current run.

After all enqueued file copies have completed in a normal run, SnapshotStore
uploads each updated local temporary snapshot back to its peer through snapshot
SWAP staging. Before upload, it commits or rolls back every transaction,
finalizes every statement, cursor, and reader, and closes every SQLite
connection to that local `snapshot.db`. The transport upload reads the closed
local file, not a live SQLite connection, and the peer receives only
`snapshot.db`, not SQLite sidecar files. The uploaded database must be usable
as a self-contained SQLite database.

Normal snapshot upload writes the closed local database to
`.kitchensync/SWAP/snapshot.db/new`, closes that peer-side `new` file, moves an
existing live `.kitchensync/snapshot.db` to
`.kitchensync/SWAP/snapshot.db/old`, moves `new` into live
`.kitchensync/snapshot.db`, and then removes `old`. This sequence must replace
an existing peer snapshot on transports whose `rename(src, dst)` rejects an
existing destination. If upload fails before `old` exists, SnapshotStore
reports the failure without removing the live snapshot; any leftover `new` is
handled by the next normal startup recovery. If upload fails after `old`
exists, SnapshotStore leaves the peer-side snapshot SWAP state for the next
normal startup recovery. When overlapping normal runs upload snapshots to the
same peer, the peer-side live snapshot reflects the last completed snapshot
upload.

## Boundaries

SnapshotStore does not parse command-line arguments, normalize peer URLs,
select fallback URLs, authenticate SFTP sessions, create peer roots, decide
canon or subordinate roles, make sync decisions, enforce copy concurrency,
copy user files, displace user files to `BAK/`, recover user-data SWAP
directories, clean peer `BAK/` or `TMP/` directories, set filesystem
modification times, or format stdout.

SnapshotStore does not create BAK directories, create TMP directories, or write
log lines. It only provides the timestamp string generator those callers use
when they need spec-format timestamp values.

SnapshotStore does not decide when a peer is removed from the global reachable
set. It reports structured snapshot startup failures for recovery and download
errors so the caller can exclude the peer, re-check startup conditions, and
print the required diagnostics.

SnapshotStore does not upload snapshots during dry-run. Dry-run callers may use
the local temporary databases exactly like a normal run, but peer-side snapshot
SWAP recovery and peer-side upload are outside this child in dry-run mode
because dry-run must not create, modify, rename, delete, displace, or upload
peer files.

SnapshotStore does not own the moment when final upload begins. The caller must
wait for all enqueued file copies to finish before invoking normal snapshot
upload. Once invoked, SnapshotStore guarantees that the local database file is
closed and sidecar-free before handing bytes to transport upload.

## Invariants

- Peer snapshot state lives at `.kitchensync/snapshot.db`.
- Snapshot SWAP state lives only under `.kitchensync/SWAP/snapshot.db/`.
- SQLite databases this child creates or updates use rollback-journal mode.
- Only the closed `snapshot.db` file is uploaded as peer snapshot state.
- The schema is exactly one `snapshot` table, no views, the seven specified
  columns, and indexes on `parent_id`, `last_seen`, and `deleted_time`.
- The sync root directory has no row.
- Path IDs are deterministic 11-character base62 xxHash64 seed-0 strings.
- Non-NULL snapshot timestamp fields use
  `YYYY-MM-DD_HH-mm-ss_ffffffZ`.
- Generated timestamps are process-local monotonic and not reused for
  `last_seen` writes within a run.
- Confirmed absences and displacement cascades copy deletion estimates instead
  of generating deletion timestamps.
